# VibeFlow — Glossary

Plain-English definitions for technical terms used in VibeFlow.

---

**Approval Tier**
The level of review required before an action executes. Tier 1 = automatic. Tier 2 = reviewed by a second AI model. Tier 3 = requires human approval. See `/docs/approval-policy.md`.

**Artifact**
A file or document generated by the system. Examples: a handoff document, a diff file, a build log.

**Audit Log**
A permanent record of every major action taken in the system. Each entry shows who did what, when, which AI model was involved, and whether it was approved.

**Broadcast Channel**
A real-time communication channel in Supabase used for fast, ephemeral messages (like heartbeats and status updates) that don't need to be stored permanently in the database.

**Build Metadata**
Information about a specific build of the app: version number, git commit SHA, commit date, and release channel. Injected automatically at build time and shown in the top bar.

**Commit SHA**
A unique identifier for a specific git commit. Looks like `abc1234`. Shown in the top bar so you always know exactly which version of the app you are running.

**Context Window**
The maximum amount of text an AI model can "see" at once. When a conversation gets very long, it may exceed the context window, causing the AI to lose track of earlier messages. The handoff system helps manage this.

**Conversation**
A chat thread inside a project. The user primarily talks to the Orchestrator in a conversation. Conversations are saved and synced to the cloud.

**Coolify**
An open-source platform for deploying and managing web applications. VibeFlow integrates with Coolify to trigger deployments, view logs, and manage running apps.

**Deploy Target**
The server or platform where an application is deployed and runs. In VibeFlow, Coolify is the primary deploy target.

**Device**
A computer where VibeFlow is installed and signed in. Each device is registered with a name and last-seen timestamp. Only one device can actively drive a conversation at a time.

**Device Ownership**
Which device is currently actively running a conversation. The owning device has the execution lease. Other devices can watch in read-only mode.

**Diff**
A visual comparison showing what changed in a file. Green lines were added; red lines were removed. VibeFlow shows diffs in the right panel when the AI makes code changes.

**Execution Lease**
A temporary claim that a device holds to actively run a conversation. The device renews the lease every 15 seconds with a heartbeat. If the heartbeat stops, the lease expires and another device can take over.

**Execution Stream**
The left panel in VibeFlow. Shows what the AI is doing right now: which Mode is active, what tools are being called, what files are being changed, and summaries of completed steps.

**GHCR (GitHub Container Registry)**
GitHub's service for storing Docker images. In the Albert DevOps template, GitHub Actions builds a Docker image and pushes it to GHCR, then Coolify pulls it from there.

**Git**
A version control system that tracks changes to code over time. VibeFlow uses git for managing code changes, commits, and pushes.

**GitHub Actions**
GitHub's built-in automation system. Used in VibeFlow to automatically build Docker images, run tests, and trigger deployments when code is pushed.

**GitHub Secrets**
A secure place to store sensitive values (like API keys) that GitHub Actions needs to run. These are separate from the secrets stored in VibeFlow itself.

**Handoff**
A one-click operation that generates a complete summary package for a brand-new AI session to continue your project without confusion. See `/docs/handoff-process.md`.

**Heartbeat**
A signal sent every 15 seconds by the device that owns an active conversation run. Proves the device is still alive and working. If heartbeats stop, the run is marked as recoverable.

**Idiosyncrasy**
An intentional code or architecture choice that looks wrong or unusual but was done for a good reason. Documented in `/docs/idiosyncrasies.md`.

**IPC (Inter-Process Communication)**
The mechanism Electron uses to let the UI (renderer process) talk to the local machine capabilities (main process). For example, when the AI wants to read a file, the UI sends an IPC message to the main process, which reads the file and sends the contents back.

**keytar**
A Node.js library that stores secrets in the operating system's secure storage. On Windows, this is the Windows Credential Manager. Used to store API keys and other sensitive values without putting them in plain text files.

**Lease**
See "Execution Lease."

**MCP (Model Context Protocol)**
A standard for connecting AI models to external tools and data sources. VibeFlow supports MCP server connections so AI Modes can use external tools.

**Mode**
A specialist AI persona in VibeFlow. Each Mode has a name, a set of instructions (soul), a specific AI model assigned to it, and a set of tools it can use. Examples: Orchestrator, Coder, Debugger, DevOps.

**Mode Soul**
The detailed instructions that define how a Mode behaves. Think of it as the Mode's personality and job description. Editable in the GUI.

**Monorepo**
A single code repository that contains multiple related packages or applications. VibeFlow uses a monorepo with `apps/desktop` (the Electron app) and `packages/*` (shared modules).

**OpenRouter**
A service that provides access to many AI models (GPT-4, Claude, Gemini, etc.) through a single API. VibeFlow uses OpenRouter as its primary AI provider.

**Orchestrator**
The main AI Mode that the user talks to. The Orchestrator sees the big picture, delegates work to specialist Modes, and reports back to the user. It is the center of gravity in VibeFlow.

**pnpm**
A fast, efficient package manager for Node.js. Used in VibeFlow's monorepo to manage dependencies across all packages.

**Project**
A container in VibeFlow for a specific piece of work. A project has conversations, attached files/repos, DevOps settings, and Mode configurations.

**Provenance**
A record of where something came from. In VibeFlow, every action has provenance: which Mode requested it, which AI model executed it, which conversation it belongs to, and when it happened.

**Realtime**
Supabase's feature for pushing live updates to all connected clients. Used in VibeFlow to show conversation updates, run state changes, and ownership changes on all devices instantly.

**Recoverable**
A run state meaning the owning device missed heartbeats and the run is now available for another device to take over. The run is not lost — it can be resumed.

**Release Channel**
Which version stream the app is on. Examples: `stable` (tested, reliable releases) or `beta` (newer features, may have bugs).

**Row-Level Security (RLS)**
A Supabase/PostgreSQL feature that ensures users can only access their own data. Even if someone had the database connection string, they could not read another user's data.

**Run State**
The current status of a conversation run. Examples: idle, running, waiting_for_human_approval, failed, completed, recoverable. Visible in the UI and synced across all devices.

**Second-Model Review**
A Tier 2 approval where a fast, cheap AI model reviews a proposed action and either approves it automatically or escalates it to the human. Designed to reduce human interruption.

**Soul**
See "Mode Soul."

**Sprint**
A named phase of development work with a specific set of steps and goals. VibeFlow tracks the current sprint in `CURRENT_TASK.md` and `/docs/owner-timeline.md`.

**SQLite**
A lightweight database that runs locally on the device. Used in VibeFlow as a local cache for synced data, so the app works fast and can function offline.

**Supabase**
A hosted service that provides a database (Postgres), authentication, real-time updates, and file storage. Used as VibeFlow's cloud backend.

**Sync**
The process of keeping data consistent between the local device and the cloud. VibeFlow syncs projects, conversations, Mode settings, and more to Supabase.

**TypeScript**
A programming language that adds type safety to JavaScript. Used throughout VibeFlow for more reliable, readable code.

**Vite**
A fast build tool for web applications. Used in VibeFlow to build the React UI quickly during development and for production releases.
