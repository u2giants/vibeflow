## 4. Guidance for the Next Architect / Orchestrator

1. The repo is dense and evolving fast. Before trusting any `.md` file, check its "Last updated" date against commit history.
2. `apps/desktop/src/lib/` is the authoritative source for application code. `packages/*` is canonical for shared-types, storage, and build-metadata only; for everything else, treat `packages/*/README.md` as historical aspiration, not current architecture.
3. `main/index.ts` will keep growing until someone does the split described in [`docs/what-is-left.md`](what-is-left.md:1). Every new IPC handler adds to the monolith.
4. Every future change that adds brownfield-column weirdness, dual schemas, race-condition-dependent ordering, or packaging-vs-dev path switches MUST add an entry to [`docs/idiosyncrasies.md`](idiosyncrasies.md:1) in the same commit.
5. The rebuild spec in `rebuild/` is the binding design source of truth for Components 10–22. The component-analysis files (`*_COMPONENT_*_IMPLEMENTATION_ANALYSIS.md`) are binding implementation plans that Builder has already executed; do not re-plan without Orchestrator approval.
