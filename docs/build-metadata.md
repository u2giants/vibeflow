# VibeFlow — Build Metadata

Last updated: 2026-04-12

---

## What Is Build Metadata?

Build metadata is information about a specific build of the VibeFlow app:
- **Version number** — e.g., `1.2.0`
- **Git commit SHA** — e.g., `abc1234` (the exact code snapshot this build was made from)
- **Commit date/time** — when that commit was made
- **Release channel** — e.g., `stable` or `beta`

This information is shown in the top bar of the app so you always know exactly what version you are running.

---

## Where It Is Displayed

The top bar always shows:
```
VibeFlow v1.2.0  |  commit: abc1234  |  2026-04-11  |  stable  |  albert@example.com  |  🟢 Synced
```

---

## How It Gets Injected

Build metadata is injected automatically at build time by a script. It is never hardcoded manually.

**The injection script:** [`scripts/inject-build-metadata.js`](../scripts/inject-build-metadata.js)

**What the script does:**
1. Reads the version from `package.json`
2. Runs `git rev-parse --short HEAD` to get the commit SHA
3. Runs `git log -1 --format=%ci` to get the commit date
4. Reads the release channel from an environment variable (`RELEASE_CHANNEL`, defaults to `stable`)
5. Writes all four values to a generated file: `apps/desktop/src/lib/build-metadata/generated.ts`

**The generated file looks like:**
```typescript
// AUTO-GENERATED — DO NOT EDIT MANUALLY
// Generated by scripts/inject-build-metadata.js at build time
export const BUILD_METADATA = {
  version: '1.2.0',
  commitSha: 'abc1234',
  commitDate: '2026-04-11T17:00:00Z',
  releaseChannel: 'stable',
} as const;
```

**The generated file is:**
- Listed in `.gitignore` (not committed to git — it is generated fresh each build)
- Imported by `apps/desktop/src/lib/build-metadata/index.ts` with a try/catch fallback
- If the file doesn't exist yet, safe defaults are used (`0.0.0-dev`, `unknown`, `dev`)
- Used to populate the top bar

---

## When the Script Runs

| Scenario | When metadata is injected |
|---|---|
| Development mode (`npm run dev`) | Script runs at dev server start; uses current git state |
| Production build (`npm run build`) | Script runs before the build; uses current git state |
| GitHub Actions release build | Script runs as a build step; `RELEASE_CHANNEL` is set by the workflow |

---

## Release Channels

| Channel | Meaning |
|---|---|
| `stable` | Tested, reliable releases for normal use |
| `beta` | Newer features, may have minor bugs |
| `dev` | Local development builds (not distributed) |

The release channel is set by the `RELEASE_CHANNEL` environment variable when building. In GitHub Actions, the release workflow sets this automatically based on the git tag or branch.

---

## Why Not Hardcode It?

Hardcoding version numbers and commit SHAs is error-prone and easy to forget. Injecting them automatically at build time means:
- The top bar always shows accurate information
- No human has to remember to update a version number
- Every build is traceable to an exact git commit
- Rollback is easy — you can always identify which build you are running

---

## Auto-Update Integration

The version number in build metadata is also used by `electron-updater` to check for updates. When the app starts (in packaged builds only), it compares the current version against the latest GitHub Release. If a newer version exists, a non-intrusive banner appears below the top bar.

**How auto-update works:**
1. `electron-updater` checks GitHub Releases 5 seconds after app startup
2. If a newer version is found, the `UpdateBanner` component shows "Update available: v{version}"
3. User clicks "Install Now" → the update downloads in the background
4. When downloaded, the banner shows "Update ready — Restart to apply"
5. User clicks "Restart Now" → the app installs the update and restarts

**Safety guarantees:**
- Auto-updater only runs in packaged builds (not dev mode)
- All updater errors are non-fatal — the app never crashes if update checks fail
- `autoDownload = false` — the user decides when to download, no surprise downloads

See [`/docs/release-process.md`](release-process.md) for the full release and update flow.
